# 노트북 4. 컨텍스트 윈도우와 토큰

> Phase 2 — 제어

토큰은 LLM의 화폐입니다. 비용도 토큰, 성능 한계도 토큰, 응답 품질도 토큰에 달렸습니다.

**학습 목표**
- 토큰의 실체(서브워드 단위)를 이해하고 직접 측정할 수 있다
- 한국어와 영어의 토큰 효율 차이를 설명할 수 있다
- 컨텍스트 윈도우의 개념과 한계를 이해한다
- 멀티턴 대화에서의 토큰 누적과 비용 구조를 계산할 수 있다

## 토큰의 실체

LLM은 텍스트를 단어 단위가 아니라 **토큰(token)** 단위로 처리합니다. 토큰은 **서브워드(subword)** 단위로, 하나의 단어가 여러 토큰으로 쪼개질 수도 있고, 짧은 단어는 하나의 토큰이 될 수도 있습니다.

- `"hello"` — 1토큰
- `"unhappiness"` — `["un", "happiness"]` — 2토큰
- `"인공지능"` — 여러 토큰 (한국어는 더 많은 토큰 소비)

> 핵심: 토큰은 LLM이 텍스트를 이해하는 최소 단위입니다. 비용, 속도, 컨텍스트 한계 모두 토큰 수로 결정됩니다.

텍스트 유형에 따라 토큰화 효율이 달라집니다. 일반 텍스트, 코드, JSON, URL, 반복 텍스트는 각각 다른 글자/토큰 비율을 보입니다. 코드에서는 식별자와 기호가 개별 토큰으로 처리되고, URL에서는 경로 구분자와 쿼리 파라미터가 별도 토큰이 됩니다.

### count_tokens() API

Gemini는 `client.models.count_tokens()` API로 텍스트의 토큰 수를 비용 없이 사전 측정할 수 있습니다.

```python
result = client.models.count_tokens(
    model="gemini-2.5-flash",
    contents="Hello, world!",
)
print(result.total_tokens)  # 4
```

`system_instruction`을 포함한 전체 입력의 토큰 수도 측정할 수 있습니다. 실제 API 호출 형태와 동일하게 `config` 파라미터에 system prompt를 전달하면 됩니다.

```python
tokens = client.models.count_tokens(
    model="gemini-2.5-flash",
    contents="파이썬이란?",
    config={"system_instruction": "간결하게 답변하세요."},
).total_tokens
```

전달 방식(문자열, `Content` 객체, 리스트 감싸기)에 관계없이 동일한 텍스트는 동일한 토큰 수를 반환합니다.

### 한국어 vs 영어 토큰 효율

한국어는 영어보다 토큰 효율이 낮습니다. 유니코드 다바이트 문자와 조합형 특성 때문에, 같은 의미를 전달하는 데 약 **1.5~2배**의 토큰을 소비합니다. 이는 한국어 서비스 운영 시 비용과 컨텍스트 윈도우 측면에서 불리하게 작용합니다.

### 토큰 수 어림 규칙

API 호출 없이 토큰 수를 빠르게 추정하는 경험 규칙입니다.

| 언어 | 어림 규칙 | 근거 |
|------|----------|------|
| 영어 | 1토큰 ≈ 4글자 (또는 0.75단어) | 라틴 알파벳 기반 서브워드 |
| 한국어 | 1토큰 ≈ 1~2글자 | 유니코드 다바이트 문자 |
| 코드 | 단어 수와 비슷 | 식별자, 기호가 개별 토큰 |

정확하지는 않지만, 비용이나 컨텍스트 사용량을 빠르게 어림할 때 유용합니다.

## 컨텍스트 윈도우

**컨텍스트 윈도우(Context Window)**는 모델이 한 번에 처리할 수 있는 최대 토큰 수입니다. 입력 토큰과 출력 토큰의 합이 이 한계를 넘을 수 없습니다.

```
입력 토큰 + 출력 토큰 <= 컨텍스트 윈도우
```

| 모델 | 컨텍스트 윈도우 | 최대 출력 토큰 |
|------|----------------|---------------|
| gemini-2.5-flash | 1,048,576 (약 100만) | 65,536 |
| gemini-2.5-pro | 1,048,576 (약 100만) | 65,536 |
| gemini-2.0-flash | 1,048,576 (약 100만) | 8,192 |

100만 토큰은 약 70만 단어, 책 약 10권 분량에 해당합니다. 입력이 윈도우를 초과하면 API가 에러를 반환합니다.

> 핵심: 컨텍스트 윈도우가 크다고 해서 항상 좋은 것은 아닙니다. 긴 컨텍스트에서는 모델의 주의력이 분산되는 **Lost in the Middle** 현상이 발생할 수 있습니다. 중간에 위치한 정보는 앞이나 뒤에 있는 정보보다 덜 잘 인식됩니다.

### usage_metadata로 사용량 확인

API 응답의 `usage_metadata`에서 정확한 토큰 사용량을 확인할 수 있습니다.

```python
response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents="대한민국의 역사를 간략히 설명해주세요.",
)
usage = response.usage_metadata
print(usage.prompt_token_count)      # 입력 토큰
print(usage.candidates_token_count)  # 출력 토큰
print(usage.total_token_count)       # 입력 + 출력
```

Thinking이 활성화된 경우 `thoughts_token_count` 필드에 추론 토큰 수가 추가로 포함됩니다.

## 비용 구조

Gemini API 비용은 토큰 단위로 과금됩니다. **입력 토큰보다 출력 토큰의 단가가 높습니다.**

### Gemini 모델별 가격 비교

| 모델 | 입력 ($/1M 토큰) | 출력 ($/1M 토큰) | Flash 대비 |
|------|-----------------|-----------------|----------|
| gemini-2.5-flash | $0.15 | $0.60 | 1x |
| gemini-2.5-pro | $1.25 | $10.00 | ~12x |
| gemini-2.0-flash | $0.10 | $0.40 | ~0.7x |

gemini-2.5-flash 기준 200K 토큰을 초과하는 입력은 단가가 2배로 올라갑니다 (입력 $0.30, 출력 $1.20).

비용 계산의 핵심 구조는 다음과 같습니다.

```python
def calculate_cost(input_tokens, output_tokens,
                   input_price_per_million=0.15,
                   output_price_per_million=0.60):
    input_cost = (input_tokens / 1_000_000) * input_price_per_million
    output_cost = (output_tokens / 1_000_000) * output_price_per_million
    return input_cost + output_cost
```

단일 호출은 매우 저렴합니다. 그러나 멀티턴 대화에서는 상황이 달라집니다.

### 멀티턴 대화의 토큰 누적

멀티턴 대화에서 각 턴마다 **이전 전체 이력이 입력에 포함**됩니다. 10턴 대화의 비용은 "1턴 비용 x 10"이 아니라, 이력 누적으로 인해 훨씬 더 많습니다.

매 턴 user 30토큰, AI 100토큰, system 20토큰으로 가정한 시뮬레이션에서, 턴별 입력 토큰은 1턴째 50토큰에서 20턴째 2,500토큰 이상으로 증가합니다. 20턴 시점의 실제 누적 비용은 선형 가정(매 턴 동일 비용)의 약 10배에 달합니다.

> 핵심: 멀티턴 대화의 비용은 턴 수에 따라 **이차(quadratic)**적으로 증가합니다. 이것이 컨텍스트 매니지먼트(이력 관리 전략)가 필요한 이유입니다.

### Thinking 토큰

Gemini 2.5 모델의 **Thinking(추론)** 기능이 활성화되면, 모델이 답변 전에 내부적으로 "생각"하는 과정에서 추가 토큰이 소비됩니다.

| 토큰 종류 | 설명 | 과금 |
|----------|------|------|
| Input tokens | 사용자 입력 + system prompt | 입력 단가 |
| Output tokens | 실제 응답 텍스트 | 출력 단가 |
| Thinking tokens | 내부 추론 과정 | 출력 단가와 동일 |

`thinking_budget`을 높게 설정하면 추론 토큰이 크게 증가할 수 있습니다. 단순한 질문에는 thinking을 비활성화(`thinking_budget=0`)하는 것이 비용 효율적입니다.

## Long Context vs RAG

Gemini의 100만 토큰 윈도우 덕분에 문서를 통째로 프롬프트에 넣을 수 있습니다. 하지만 모든 상황에서 그것이 최선은 아닙니다. 문서를 활용하는 두 가지 전략을 비교합니다.

| 기준 | Long Context | RAG |
|------|-------------|-----|
| 구현 복잡도 | 낮음 (그냥 넣으면 됨) | 높음 (임베딩 + 벡터DB + 검색) |
| 비용 (호출당) | 높음 (전체 문서 토큰) | 낮음 (검색된 청크만) |
| 정확도 | 문서 전체를 볼 수 있음 | 검색 품질에 의존 |
| 속도 | 느림 (긴 입력 처리) | 빠름 (짧은 입력) |
| Lost in the Middle | 발생 가능 | 덜 발생 (짧은 컨텍스트) |
| 문서 업데이트 | 즉시 반영 | 재인덱싱 필요 |

50,000토큰 문서에 대해 100회 질문하는 시뮬레이션에서, RAG(검색 청크 2,000토큰)는 Long Context 대비 약 96%의 비용을 절감합니다. 문서가 클수록 이 차이는 더 벌어집니다.

> 핵심: 문서가 짧고(10페이지 이하) 호출 빈도가 낮으면 Long Context가 간편합니다. 문서가 길거나 호출이 빈번하면 RAG가 비용 효율적입니다.

---

## 정리

- **토큰**은 서브워드 단위이며, 한국어는 영어 대비 1.5~2배의 토큰을 소비한다
- `count_tokens()` API로 비용 없이 토큰 수를 사전 측정할 수 있고, `usage_metadata`로 실제 사용량을 확인한다
- 멀티턴 대화의 비용은 이력 누적으로 인해 이차적으로 증가하므로, 컨텍스트 관리 전략이 필수적이다
- 모델 선택(Flash vs Pro)에 따라 동일 호출의 비용이 최대 12배까지 차이가 난다
- Long Context는 간편하지만 비용이 높고, RAG는 구현이 복잡하지만 비용 효율적이다 — 문서 크기와 호출 빈도에 따라 선택한다
