# 노트북 14. 챗봇 평가 지표 및 방법

> Phase 5 — 품질 관리

"잘 되는 것 같다"는 평가가 아닙니다. 체계적이고 반복 가능한 평가 체계가 있어야 무엇이 좋고 무엇이 부족한지 파악하고, 근거를 기반으로 개선할 수 있습니다.

**학습 목표**
- LLM 평가가 어려운 이유를 이해한다
- 정확성, 관련성, 충실도, 유해성, 톤 등 평가 기준을 설계할 수 있다
- LLM-as-Judge 패턴으로 자동 평가를 구현할 수 있다
- Pairwise 비교와 RAG Triad 평가를 수행할 수 있다
- LangSmith로 체계적인 평가 파이프라인을 구성할 수 있다

## 왜 LLM 평가가 어려운가

전통적인 소프트웨어는 `assert result == expected`로 테스트할 수 있지만, LLM은 근본적으로 다릅니다.

| 구분 | 전통 소프트웨어 | LLM |
|------|----------------|-----|
| 정답 | 하나 | 좋은 답변이 여러 개 |
| 출력 특성 | 결정적 | 확률적 (같은 입력, 다른 결과) |
| 테스트 방법 | 단위 테스트 | 주관적 품질 판단 필요 |
| 결과 판정 | 통과/실패 | 스펙트럼 (얼마나 좋은가?) |

이러한 특성 때문에 평가에는 세 가지 접근이 존재합니다.

| 접근 | 방법 | 장점 | 단점 |
|------|------|------|------|
| 인간 평가 | 사람이 직접 채점 | 가장 신뢰성 높음 | 비용 높음, 일관성 낮음 |
| 규칙 기반 | 키워드, 길이, 형식 체크 | 빠르고 저렴 | 의미적 품질 판단 불가 |
| **LLM-as-Judge** | LLM에게 루브릭 주고 채점 | 자동화, 확장 가능 | Judge 편향 가능 |

> 실무에서는 세 가지를 조합합니다. 규칙 기반으로 기본 필터링 후, LLM-as-Judge로 대량 평가하고, 인간이 샘플을 검수하는 단계적 접근이 일반적입니다.

## 평가 기준 설계

"좋은 답변"을 판단하려면 먼저 **무엇이 좋은 건지** 정의해야 합니다. 주요 평가 기준은 다음과 같습니다.

| 기준 | 질문 | 예시 |
|------|------|------|
| **정확성**(Correctness) | 사실적으로 맞는가? | "서울은 한국의 수도" (O) |
| **관련성**(Relevance) | 질문에 대한 답인가? | Q: 날씨? A: 경제 이야기 (X) |
| **충실도**(Faithfulness) | 주어진 문서에 근거하는가? | 문서에 없는 내용을 지어냄 (X) |
| **유해성**(Harmfulness) | 위험/부적절한 내용이 있는가? | 폭력적 표현 (X) |
| **톤**(Tone) | 챗봇 페르소나에 맞는가? | 공식 챗봇인데 반말 (X) |

> 모든 기준을 동시에 평가할 필요는 없습니다. 서비스 특성에 맞는 핵심 기준 2~3개를 선택하는 것이 효과적입니다.

## 규칙 기반 평가

LLM 호출 없이 코드만으로 체크할 수 있는 기본 품질 지표입니다. 비용이 들지 않으므로 첫 번째 필터로 활용합니다.

```python
def eval_not_empty(response: str) -> dict:
    """응답이 비어있지 않은지 확인"""
    return {"metric": "not_empty", "score": 1.0 if len(response.strip()) > 0 else 0.0}

def eval_contains_keywords(response: str, keywords: list[str]) -> dict:
    """응답에 필수 키워드가 포함되어 있는지 확인"""
    found = [kw for kw in keywords if kw in response]
    score = len(found) / len(keywords) if keywords else 0.0
    return {"metric": "keywords", "score": score, "found": found}
```

규칙 기반 평가는 빠르고 저렴하지만, "답변이 정확한가", "맥락에 맞는가" 같은 의미적 품질은 판단할 수 없습니다. 기본 필터로 사용하고, 의미적 평가는 LLM-as-Judge에 맡깁니다.

## LLM-as-Judge 패턴

**LLM-as-Judge**는 LLM에게 평가 기준(루브릭)을 주고, 다른 LLM의 응답을 채점하게 하는 패턴입니다. 질문과 응답, 그리고 평가 기준을 Judge 프롬프트에 넣으면 점수와 근거를 함께 반환받습니다.

### Judge 프롬프트 설계

좋은 Judge의 핵심은 **구체적인 루브릭**입니다.

| 원칙 | 설명 |
|------|------|
| 명확한 기준 | 각 점수가 의미하는 바를 구체적으로 정의 |
| 점수 스케일 | 1~5 또는 1~10 (너무 세밀하면 일관성 저하) |
| 근거 요구 | 점수뿐 아니라 이유(reasoning)도 함께 출력 |
| 구조화 출력 | JSON 형식으로 파싱 가능하게 |

나쁜 루브릭: `"정확성을 1~5점으로 평가하세요"` -- 기준이 모호합니다.

좋은 루브릭: `"5: 모든 사실이 정확, 4: 대부분 정확하지만 사소한 부정확함, 3: 핵심은 맞지만 일부 틀림, 2: 틀린 정보가 많음, 1: 완전히 틀림"` -- 각 점수의 의미가 명확하여 Judge의 일관성이 높아집니다.

```python
judge_prompt = ChatPromptTemplate.from_template(
    """아래 질문에 대한 AI 응답을 평가하세요.
[질문] {question}
[AI 응답] {response}
[평가 기준]
- 정확성 (1~5): 사실적으로 올바른가?
- 관련성 (1~5): 질문에 대한 답변인가?
- 완성도 (1~5): 충분히 상세한가?
JSON 형식으로 답하세요: {{"accuracy": {{"score": N, "reason": "..."}}, ...}}"""
)
judge_chain = judge_prompt | llm | StrOutputParser()
```

> JSON 파싱 오류를 줄이려면 `with_structured_output(PydanticModel)`을 사용하여 Pydantic 모델로 결과를 받는 것이 안정적입니다(노트북 8 참고).

### Judge 모델 선택

| 전략 | 설명 | 장점 | 단점 |
|------|------|------|------|
| 동일 모델 | 생성 모델과 같은 모델로 평가 | 비용 절감 | 자기 선호 편향 |
| 상위 모델 | 더 강력한 모델로 평가 | 높은 정확도 | 비용 증가 |
| 다중 Judge | 여러 모델로 평가 후 합산 | 편향 감소 | 비용 높음 |

생성 모델보다 같거나 상위 모델을 Judge로 사용하는 것이 권장됩니다. 예를 들어 Flash로 생성하고, Flash 또는 Pro로 평가합니다.

### 평가 방법 선택 가이드

정답이 있는 경우(**참조 기반 평가**)와 없는 경우(**참조 없는 평가**)로 나뉩니다. 참조 기반은 정확성 평가에, 참조 없는 평가는 유창성/톤 평가에 적합합니다. 정답이 있다면 규칙 기반(키워드, 형식)이나 임베딩 유사도(노트북 12)를 활용할 수 있고, 정답이 없다면 LLM-as-Judge가 의미적 품질을 판단합니다.

## Pairwise 비교

**Pairwise 비교**는 두 응답을 나란히 놓고 "어느 것이 더 나은가"를 판정하는 방식입니다. 절대 점수를 매기기 어려운 경우에도 상대적 비교는 수행할 수 있어, A/B 테스트나 모델 교체 판단에 유용합니다.

```python
pairwise_prompt = ChatPromptTemplate.from_template(
    """[질문] {question}
[응답 A] {response_a}
[응답 B] {response_b}
어느 응답이 더 나은지 JSON으로 답하세요:
{{"winner": "A" 또는 "B" 또는 "TIE", "reason": "판단 근거"}}"""
)
```

### Judge 편향 주의

Pairwise 비교에서는 두 가지 편향을 주의해야 합니다.

- **위치 편향**(Position Bias): Judge가 A를 먼저 읽기 때문에 A를 선호하는 경향이 있습니다. A/B 순서를 바꿔 두 번 평가하고 결과가 일관적인지 검증해야 합니다.
- **길이 편향**(Length Bias): 같은 정확도라면 긴 답변이 선호되는 경향이 있습니다. 항상 상세한 답변이 좋은 것은 아니므로 주의가 필요합니다.

## RAG 특화 평가: RAG Triad

RAG 시스템(노트북 13)은 검색된 문서를 기반으로 답변을 생성하므로, 일반 평가 외에 **RAG Triad**라는 3가지 특화 지표가 필요합니다.

| 지표 | 평가 대상 | 질문 |
|------|----------|------|
| **Context Relevance** | 검색된 문서 | 검색된 문서가 질문과 관련 있는가? |
| **Answer Faithfulness** | 답변 vs 문서 | 답변이 검색된 문서에 근거하는가? |
| **Answer Relevance** | 답변 vs 질문 | 답변이 질문에 대한 답인가? |

세 지표의 관계를 도식화하면 다음과 같습니다:

```
질문 ←── Answer Relevance ──→ 답변
  │                              ↑
  │                    Answer Faithfulness
  │                              │
Context Relevance ──→ 검색 문서 ──┘
```

**Answer Faithfulness**는 **할루시네이션**(Hallucination) 감지의 핵심입니다. 답변이 문서에 없는 정보를 포함하면 Faithfulness가 낮아집니다. 예를 들어 문서에 `sort()`와 `sorted()`만 언급되어 있는데 답변이 `numpy.sort()`나 `pandas.sort_values()`를 언급하면, 비록 사실일지라도 문서에 근거하지 않은 정보이므로 감점됩니다.

> 세 지표 중 어느 것이 낮은지에 따라 개선 방향이 달라집니다. Context Relevance가 낮으면 검색 품질을, Faithfulness가 낮으면 프롬프트의 문서 참조 제약을, Answer Relevance가 낮으면 프롬프트의 질문 집중도를 개선해야 합니다.

## LangSmith Evaluation

**LangSmith**는 LangChain 생태계의 평가 플랫폼으로, 테스트 케이스 관리부터 자동 평가, 결과 대시보드까지 체계적인 평가 파이프라인을 제공합니다.

### 평가 파이프라인 구조

```
1. Dataset 생성   ─ 질문 + 기대 답변을 등록
2. Target 정의    ─ 평가할 체인/함수를 지정
3. Evaluator 정의 ─ 점수 계산 로직을 구현
4. evaluate() 실행 ─ 자동 평가 + 대시보드에서 결과 확인
```

```python
from langsmith.evaluation import evaluate, EvaluationResult

def not_empty_evaluator(run, example) -> EvaluationResult:
    output = run.outputs.get("answer", "")
    return EvaluationResult(key="not_empty", score=1.0 if output.strip() else 0.0)

results = evaluate(
    my_chatbot,                          # 평가 대상 함수
    data="dataset-name",                 # LangSmith 데이터셋 이름
    evaluators=[not_empty_evaluator],    # Evaluator 리스트
    experiment_prefix="demo-eval",       # 실험 이름
)
```

LangSmith 대시보드에서는 각 테스트 케이스별 점수, Evaluator별 평균, 실험 간 비교(모델 A vs B), 그리고 각 호출의 상세 트레이스를 확인할 수 있습니다.

> LangSmith가 없어도 체계적인 평가는 가능합니다. 핵심은 **데이터셋 + 평가 함수 + 결과 저장**의 3요소입니다. LangSmith는 이를 편리하게 해주는 도구일 뿐, 필수는 아닙니다.

## 오류 분석과 회귀 테스트

### 오류 분석

평가 결과에서 **낮은 점수를 받은 케이스를 분류**하는 것이 개선의 핵심입니다. "점수가 낮다"만으로는 어디를 고쳐야 하는지 알 수 없습니다.

- **사실 오류**: 잘못된 정보 포함 -- 검색 품질 또는 모델 교체 검토
- **관련성 부족**: 질문과 무관한 답변 -- 프롬프트 개선
- **불완전**: 핵심 누락 -- 검색 k값 증가 또는 프롬프트 상세화
- **할루시네이션**: 문서에 없는 내용 생성 -- Faithfulness 검증 강화

### 회귀 테스트

모델이나 프롬프트를 변경할 때, 기존 성능이 유지되는지 확인하는 것이 **회귀 테스트**(Regression Testing)입니다. 동일한 데이터셋으로 변경 전후를 비교하여 개선/유지/퇴보를 판단합니다. 평가 데이터셋을 고정해야 공정한 비교가 가능하며, 데이터셋을 자주 바꾸면 비교 기준이 사라집니다.

### 평가 비용 추정

| 평가 유형 | 추가 API 호출 | 비용 영향 |
|-----------|--------------|----------|
| 규칙 기반 | 0회 | 무료 |
| LLM-as-Judge (단일 기준) | 1회/응답 | 생성 비용의 약 50~100% |
| Pairwise | 1~2회/쌍 | 생성 비용의 2배 |
| RAG Triad | 3회/응답 | 생성 비용의 3배 |

Judge에 저렴한 모델(Flash)을 사용하면 비용을 절감할 수 있습니다.

## 전체 평가 워크플로우

```
[1단계 — 설계]  평가 기준 정의 → 루브릭 작성 → 데이터셋 구축
[2단계 — 구현]  규칙 기반 평가 → LLM-as-Judge → (RAG Triad)
[3단계 — 실행]  배치 평가 실행 → 결과 수집 → 리포트 생성
[4단계 — 분석]  오류 분류 → 개선 방안 도출 → 적용 → 재평가
[5단계 — 자동화]  CI/CD 통합 → 회귀 테스트 → 지속적 모니터링
```

> 가장 중요한 것은 평가를 한 번 하고 끝내지 않는 것입니다. 지속적으로 평가하고, 결과를 기반으로 개선하는 **피드백 루프**를 만드는 것이 체계적 품질 관리의 본질입니다.

## 정리

- LLM 평가는 전통 테스트와 달리 정답이 여러 개이고 출력이 확률적이므로, 규칙 기반 + LLM-as-Judge + 인간 검수를 단계적으로 조합하는 것이 일반적이다
- LLM-as-Judge의 품질은 루브릭 설계에 달려 있으며, 각 점수의 의미를 구체적으로 정의하고 평가 근거를 함께 출력하도록 해야 한다
- Pairwise 비교는 A/B 테스트와 모델 교체 판단에 유용하지만, 위치 편향과 길이 편향을 반드시 검증해야 한다
- RAG Triad(Context Relevance, Answer Faithfulness, Answer Relevance)는 RAG 시스템의 검색-생성 품질을 각각 진단하며, 낮은 지표에 따라 개선 방향이 달라진다
- 평가 파이프라인의 핵심은 데이터셋 + 평가 함수 + 결과 저장이며, LangSmith든 자체 구현이든 이 구조를 갖추고 지속적으로 반복하는 것이 중요하다
