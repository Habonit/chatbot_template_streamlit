## 전체 구조

모든 노트북은 **3단계**로 구성한다.

```
Part 1 — 이론     : 개념을 이해시킨다
Part 2 — 실습     : 직접 코드를 작성하며 익힌다
Part 3 — 챌린지   : 강사와 함께 응용 문제를 푼다
```

각 Part는 마크다운 헤더로 명확히 구분한다.

---

## Part 1 — 이론

### 목적
개념을 마크다운 설명과 실행 가능한 코드 예시로 가르친다.
학습자는 **읽고 실행하기만 하면** 개념을 이해할 수 있어야 한다.

### 마크다운 셀 규칙

- `##`으로 소주제를 나눈다 (노트북 제목은 `#`)
- 개념 → "왜 필요한가" → 동작 원리 순서로 설명
- 비유나 일상 언어를 적극 활용하되, 기술 용어는 처음 등장할 때 **굵게** 표시하고 한 줄로 정의
- 핵심 포인트는 `> ` 인용 블록으로 강조
- 코드 셀 직전에 "아래 코드는 ~를 보여줍니다" 형태로 **무엇을 확인해야 하는지** 안내

### 코드 셀 규칙

- 모든 코드 셀은 **실행 가능해야** 한다 (의존성 설치 셀 포함)
- 주석은 "왜"를 설명한다. "무엇"은 코드 자체가 말하게 한다
- 출력 결과가 학습에 중요하면 `print()`로 명시적으로 출력
- 한 셀에 하나의 개념만. 셀이 길어지면 분리

### 이론 셀 예시

마크다운 셀:
```markdown
## Temperature란?

LLM이 다음 토큰을 선택할 때, 후보 토큰들의 확률 분포를 얼마나 "뾰족하게" 만들지 결정하는 파라미터입니다.

- **temperature = 0**: 가장 확률 높은 토큰만 선택 (거의 결정적)
- **temperature = 1.0+**: 분포가 평탄해져 다양한 토큰이 선택될 수 있음

> 핵심: temperature는 "창의성 다이얼"이라고 생각하면 됩니다.
> 낮으면 정확하고 일관되지만 뻔한 답, 높으면 다양하지만 엉뚱한 답.

아래 코드는 temperature 0.0과 1.5에서 동일한 프롬프트의 결과가 어떻게 달라지는지 보여줍니다.
```

코드 셀:
```python
# temperature에 따른 응답 차이 비교
# 같은 프롬프트를 두 번 호출하되, temperature만 다르게 설정

prompt = "인공지능의 미래를 한 문장으로 예측해줘"

# temperature=0.0: 거의 결정적 — 매번 비슷한 답
response_low = client.models.generate_content(
    model="gemini-2.5-flash",
    contents=prompt,
    config={"temperature": 0.0},
)
print(f"[temperature=0.0]\n{response_low.text}\n")

# temperature=1.5: 높은 무작위성 — 매번 다른 답
response_high = client.models.generate_content(
    model="gemini-2.5-flash",
    contents=prompt,
    config={"temperature": 1.5},
)
print(f"[temperature=1.5]\n{response_high.text}")
```

## Part 2 — 실습

### 목적
학습자가 **직접 코드를 작성**하며 이론에서 배운 개념을 체화한다.
정답은 주석 처리되어 있어서, 막히면 주석을 풀어 확인할 수 있다.

### 구조 규칙

- 마크다운 셀로 과제 설명 + 요구사항을 명확히 제시
- 코드 셀에 `# TODO:` 주석으로 작성할 부분을 표시
- 정답 코드는 바로 아래에 `# ---------- 정답 ----------` 구분선 + 주석 처리
- 정답 주석은 동작하는 완전한 코드여야 한다 (주석만 풀면 실행 가능)
- 검증용 `assert` 또는 `print()`를 정답 아래에 배치

### 실습 셀 예시

마크다운 셀:
```markdown
### 실습 2-1: System Prompt로 챗봇 성격 바꾸기

아래 조건을 만족하는 system prompt를 작성하세요.

**요구사항**
1. 페르소나: 조선시대 선비
2. 제약: 모든 답변을 존댓말로, 3문장 이내로
3. `system_prompt` 변수에 문자열로 저장
```

코드 셀:
```python
# TODO: 위 요구사항에 맞는 system prompt를 작성하세요
system_prompt = ""

# ---------- 정답 ----------
# system_prompt = (
#     "너는 조선시대 선비이다. "
#     "반드시 존댓말을 사용하고, "
#     "모든 답변은 3문장 이내로 작성하라."
# )

# 검증
response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents="오늘 점심 뭐 먹을까?",
    config={"system_instruction": system_prompt},
)
print(response.text)
```

### TODO 작성 규칙

- `# TODO:` 뒤에 무엇을 해야 하는지 한 줄로 명시
- 변수명, 함수명, 반환 타입 등 스캐폴딩은 미리 제공
- 학습자가 채워야 할 부분만 비워두고, 나머지 코드는 완성 상태로 제공
- 여러 단계가 필요하면 `# TODO 1:`, `# TODO 2:` 로 번호 부여

### 정답 주석 규칙

- `# ---------- 정답 ----------` 구분선으로 시작
- 모든 정답 코드 줄 앞에 `# ` 추가 (주석 해제만 하면 실행 가능)
- 정답이 여러 줄이면 전체를 주석 처리 (빈 줄 포함)
- 정답 주석 내에 추가 설명이 필요하면 `# # 설명:` 형태로 이중 주석

## Part 3 — 챌린지

### 목적
이론과 실습에서 배운 내용을 **응용**하여 스스로 설계하고 구현한다.
정답이 제공되지 않으며, **강사와 함께** 진행한다.

### 구조 규칙

- 마크다운 셀로 문제 설명 + 힌트 제공
- 코드 셀은 빈 상태 또는 최소한의 스캐폴딩만 제공
- 정답 코드 없음 — 주석 처리된 정답도 없음
- 셀 상단에 아래 안내 문구를 반드시 포함:

```markdown
> 이 챌린지는 정답이 제공되지 않습니다. 강사와 함께 진행하세요.
```

### 챌린지 설계 원칙

- 이론+실습 범위를 약간 넘어서는 응용 문제
- 정답이 하나가 아닌 열린 문제 (설계 판단이 필요한)
- 실습에서 배운 2개 이상의 개념을 조합해야 풀 수 있는 문제
- 난이도 표시: `난이도: ★☆☆`, `난이도: ★★☆`, `난이도: ★★★`

### 챌린지 셀 예시

마크다운 셀:
```markdown
### 챌린지 3-1: 나만의 컨텍스트 관리 전략 설계 (난이도: ★★☆)

> 이 챌린지는 정답이 제공되지 않습니다. 강사와 함께 진행하세요.

고객 상담 챗봇을 만든다고 가정합니다.

**조건**
- 대화가 평균 30턴 이상 이어짐
- 고객의 이름과 주문번호는 절대 잊으면 안 됨
- 비용은 최소화해야 함

**과제**
노트북 7에서 배운 4가지 전략(Sliding Window, Token 트리밍, 요약, 하이브리드) 중 어떤 조합이 최적인지 설계하고, 그 이유를 코드 주석으로 설명하세요.

**힌트**
- 단순 Sliding Window는 왜 안 되는지 먼저 생각해보세요
- system prompt에 항상 포함시켜야 할 정보는 무엇인가요?
```

코드 셀:
```python
# 여기에 자유롭게 구현하세요
```

## 추가 질문 (필수)

### 규칙
모든 Part(이론, 실습, 챌린지)가 끝날 때마다 **추가 질문 3개**를 마크다운 셀로 제시한다.
이것은 선택이 아니라 **의무**이다.

### 목적
- 이론: 방금 배운 개념을 다른 각도에서 생각해보게 만든다
- 실습: 코드를 작성하면서 놓쳤을 수 있는 포인트를 짚는다
- 챌린지: 문제를 더 깊이 확장할 수 있는 방향을 제시한다

### 질문 유형 가이드
3개 질문은 아래 유형을 골고루 섞는다:
- **이해 확인**: 개념을 자기 말로 설명할 수 있는지 ("~는 왜 그런 걸까요?")
- **비교/판단**: 두 가지 선택지 사이에서 트레이드오프를 따져보는 ("A와 B 중 어떤 상황에 어떤 것을 쓸까요?")
- **확장/적용**: 다른 상황에 적용해보거나 한계를 생각해보는 ("만약 ~라면 어떻게 달라질까요?")

### 포맷

```markdown
---

### 생각해보기

1. temperature를 0으로 설정했는데도 매번 미세하게 다른 결과가 나올 수 있을까요? 그렇다면 왜일까요?
2. 코드 생성에는 낮은 temperature, 창작에는 높은 temperature가 권장됩니다. 그렇다면 "코드를 창의적으로 리팩토링해줘"라는 요청에는 어떤 값이 적절할까요?
3. temperature와 top_p를 둘 다 최대로 올리면 어떤 일이 벌어질까요? 실제로 시도해보고 결과를 관찰해보세요.
```

### 배치 규칙
- 각 Part의 마지막 셀 바로 다음에 배치
- 헤더는 `### 생각해보기`로 통일
- 번호 매기기: 1, 2, 3 (Part마다 1부터 새로 시작)
- 질문만 제시하고, 답은 적지 않는다

## 공통 포맷 규칙

### 노트북 첫 번째 셀 (마크다운)

```markdown
# 노트북 N. 제목

> Phase X — 카테고리명

**학습 목표**
- 목표 1
- 목표 2
- 목표 3

**구성**
| Part | 내용 | 형태 |
|------|------|------|
| Part 1 — 이론 | 개념 설명 + 코드 예시 | 읽고 실행 |
| Part 2 — 실습 | TODO 과제 (정답 주석 포함) | 직접 작성 |
| Part 3 — 챌린지 | 응용 문제 (정답 없음) | 강사와 함께 |
```

### Part 구분

각 Part 시작 시 마크다운 셀로 구분선을 넣는다:

```markdown

# Part 2 — 실습

이론에서 배운 개념을 직접 코드로 작성해봅니다.
```

### 셀 번호 체계

각 Part 내에서 마크다운 헤더로 번호를 매긴다:
- 이론: `## 1.1 소주제`, `## 1.2 소주제`
- 실습: `### 실습 N-1: 제목`, `### 실습 N-2: 제목` (N = 노트북 번호)
- 챌린지: `### 챌린지 N-1: 제목 (난이도: ★☆☆)`

### 톤과 어투

- 존댓말 사용 (~합니다, ~입니다)
- 기술 용어는 영어 원문 유지하되, 처음 등장 시 한국어 설명 병기: "**Temperature**(온도 파라미터)"
- 코드 주석은 반말 허용 (간결함 우선)
- 이모지 사용 금지

## 문서 제목 컨벤션

노트북 파일명은 아래 형식을 따른다.

```
material/note_{숫자:01-99}_{제목}.ipynb
```

- 모든 노트북은 `material/` 디렉토리 내에 저장한다
- 숫자는 01부터 99까지 두 자리로 표기 (예: `01`, `02`, `15`)
- 제목은 소문자 영어 + 언더스코어 조합 (예: `temperature`, `prompt_engineering`)

**예시**
- `material/note_01_temperature.ipynb`
- `material/note_02_prompt_engineering.ipynb`
- `material/note_15_rag_basics.ipynb`
