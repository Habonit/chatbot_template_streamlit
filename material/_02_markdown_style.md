# 마크다운 문서 작성 스타일 가이드

> 이 가이드는 주피터 노트북(`.ipynb`)의 핵심 내용을 마크다운(`.md`)으로 요약하는 규칙을 정의합니다.
> 노트북을 직접 실행하지 않아도 **개념과 흐름을 이해할 수 있는 레퍼런스 문서**를 목표로 합니다.

---

## 문서의 목적

- 노트북에서 **무엇을 배우는지** 빠르게 파악할 수 있는 참고 문서
- 노트북을 열기 전 예습, 또는 학습 후 복습 용도
- 코드를 실행하지 않고도 개념을 이해할 수 있어야 함

**포함하는 것**: 개념 설명, 핵심 코드 예시, 비교표, 주요 인사이트
**포함하지 않는 것**: TODO 실습 코드, 정답 주석, 챌린지 문제, 환경 설정 셀

---

## 파일명 컨벤션

```
material/note_{숫자:01-99}_{제목}.md
```

- 대응하는 노트북과 **동일한 번호와 제목**을 사용
- 예: `note_01_api_call.ipynb` → `note_01_api_call.md`

---

## 전체 구조

모든 마크다운 문서는 아래 순서를 따른다.

```
1. 제목 + 메타 정보
2. 학습 목표
3. 개념 섹션들 (노트북 Part 1 — 이론 기반)
4. 정리
```

---

## 1. 제목 + 메타 정보

```markdown
# 노트북 N. 제목

> Phase X — 카테고리명

한두 문장으로 이 노트북의 핵심 메시지를 요약한다.
```

- 제목은 노트북 첫 셀의 제목과 동일
- 핵심 메시지는 `_00_load_map.md`의 해당 노트북 "핵심 메시지"를 기반으로 작성

---

## 2. 학습 목표

```markdown
**학습 목표**
- 목표 1
- 목표 2
- 목표 3
```

- 노트북의 학습 목표를 그대로 가져온다
- 3~4개, 동사형으로 시작 ("~할 수 있다")

---

## 3. 개념 섹션

노트북 Part 1(이론)의 내용을 **개념 단위**로 정리한다.

### 섹션 구성 규칙

- `##`으로 대주제, `###`으로 소주제를 구분
- 노트북의 섹션 번호(1.1, 1.2 등)는 사용하지 않는다 — 의미 있는 제목만 사용
- 하나의 섹션에 하나의 개념. 두 개념이 섞이면 분리

### 설명 규칙

- 개념을 먼저 설명하고, 왜 필요한지, 어떻게 동작하는지 순서로 전개
- 기술 용어는 처음 등장 시 **굵게** 표시하고 간단히 정의
- 핵심 포인트는 인용 블록으로 강조:

```markdown
> 핵심: LangChain의 가장 큰 가치는 모델 교체 용이성입니다.
```

- 비교가 필요한 내용은 **표**를 적극 활용

### 코드 예시 규칙

- 개념을 설명하는 데 **필수적인 코드만** 포함
- 노트북의 코드를 그대로 복사하지 않는다 — 핵심을 보여주는 **최소 코드**로 정리
- 실행 결과가 이해에 필요하면 주석으로 기대 출력을 표시:

```python
response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents="대한민국의 수도는?",
)
print(response.text)  # "서울입니다." (예시)
```

- 코드 블록 직전에 "무엇을 보여주는 코드인지" 한 줄로 설명:

```markdown
google-genai SDK의 기본 호출 구조:
```

- 동일한 개념을 여러 코드로 반복 보여주는 경우, 대표적인 코드 하나만 선택

### 포함하지 않는 코드

- 환경 설정 (`!pip install`, API 키 설정)
- 타입 확인용 코드 (`type()`, 속성 나열)만으로 구성된 셀
- 동일 패턴의 반복 호출 (batch 예시를 3번 보여줄 필요 없음)
- Part 2(실습)와 Part 3(챌린지)의 코드 전체

### 비교표 활용

노트북에서 두 가지를 비교하는 내용은 표로 정리한다:

```markdown
| 구분 | google-genai | LangChain |
|------|-------------|----------|
| 반환 타입 | `GenerateContentResponse` | `AIMessage` |
| 텍스트 접근 | `.text` | `.content` |
```

## 4. 정리

문서 마지막에 핵심 내용을 짧게 요약한다.

```markdown
---

## 정리

- google-genai는 Gemini 전용 1차 SDK, LangChain은 멀티 모델 래퍼
- 반환 객체가 다르지만 담고 있는 정보는 동일
- LCEL 체인(`prompt | model | parser`)은 이후 모든 노트북에서 반복 사용되는 핵심 패턴
```

- 3~5개 bullet point
- 새로운 정보가 아니라 문서에서 다룬 내용의 **요약**

---

## 톤과 어투

- 존댓말 사용 (~합니다, ~입니다)
- 기술 용어는 영어 원문 유지, 처음 등장 시 한국어 병기: "**LCEL**(LangChain Expression Language)"
- 이모지 사용 금지
- 간결체 — 같은 말을 반복하지 않는다

---

## 분량 기준

- 노트북 1개당 마크다운 **A4 2~4쪽** 분량 (대략 100~250줄)
- 코드 블록은 전체 분량의 **30% 이하**
- "이 내용이 없으면 노트북을 이해할 수 없는가?"를 기준으로 취사선택
