## 2. 문제점

### 2.1 비효율: 전부 켜지거나 전부 꺼짐

| 시나리오 | llm_node | casual_node | reasoning 도구 | 평가 |
|----------|----------|-------------|---------------|------|
| Thinking OFF (기본) | thinking 없음 | thinking 없음 | thinking 없음 | reasoning에 가장 필요한데 안 씀 |
| Thinking ON (토글) | thinking 있음 — **낭비** | thinking 있음 — **낭비** | thinking 있음 | 단순 대화에도 thinking 토큰 소비 |

### 2.2 기능 중복

| | reasoning 도구 (프롬프트) | Gemini Thinking (네이티브) |
|---|---|---|
| 방식 | 4단계 지시 프롬프트 | 모델 아키텍처 수준 chain-of-thought |
| 품질 | 프롬프트 의존, 불안정 | 안정적, 고품질 |
| 비용 | LLM 추가 호출 1회 전체 | thinking token만 추가 |

**이상적 구조**: 두 가지를 **reasoning 도구에서만 결합**하여 최고 품질 추론을 달성.

## 3. 목표 구조

### 3.1 LLM 인스턴스 2개 분리

```
__init__:
  self._llm                  ← thinking 없음 (llm_node, casual_node, bind_tools)
  self._llm_for_reasoning    ← thinking_budget + include_thoughts (reasoning 도구 전용)
```

> **설계 원칙**: `self._llm`은 메인 LLM으로 llm_node, casual_node, bind_tools에 사용. `self._llm_for_reasoning`은 **오직 reasoning 도구 내부 호출에서만** 사용. 다른 도구(get_current_time, web_search, search_pdf_knowledge)는 `llm` 파라미터를 사용하지 않으므로 영향 없음.

### 3.2 그래프 흐름 (변경 없음)

```
START → summary_node → router_node → casual_node → END       (self._llm)
                                    → llm_node ⇄ tool_node → END
                                      (self._llm_with_tools)
```

`tool_node` 안에서 reasoning 도구가 호출될 때만 `self._llm_for_reasoning`이 사용됨.

### 3.3 reasoning 도구 호출 시 상세 흐름

```
llm_node: "이건 reasoning 도구를 써야겠다" (thinking 없음, 도구 선택만)
  ↓
tool_node → reasoning(question, context)
  ↓ (내부)
  self._llm_for_reasoning.invoke([...])  ← thinking_budget + 프롬프트 엔지니어링
  ↓
  ReasoningResult(thought="...", analysis="...") 구조화된 반환
  ↓
llm_node: reasoning 결과를 참고하여 최종 답변 생성 (thinking 없음)
```

```python
import json

@tool
def reasoning(question: str, context: str = "") -> str:
    """복잡한 문제를 단계별로 분석합니다."""
    target_llm = reasoning_llm or llm
    if not target_llm:
        return "추론 도구가 설정되지 않았습니다."

    try:
        prompt = get_reasoning_prompt(user_input=question, context=context)
        response = target_llm.invoke([HumanMessage(content=prompt)])

        # include_thoughts=True 시 content가 list 형태
        if isinstance(response.content, list):
            thought, text = extract_thought_from_content(response.content)
            if show_thoughts and thought:
                return json.dumps(
                    {"thought": thought, "analysis": text},
                    ensure_ascii=False,
                )
            return text
        return response.content
    except Exception as e:
        # Fallback: reasoning LLM 실패 시 메인 LLM으로 재시도
        # prompt는 상위 스코프에서 이미 생성됨 → 재생성 불필요
        if target_llm is not llm and llm:
            try:
                response = llm.invoke([HumanMessage(content=prompt)])
                return response.content
            except Exception:
                pass
        return f"추론 중 오류 발생: {str(e)}"
```

**설계 결정**:

| 항목 | 이전 (매직 스트링) | 현재 (JSON) |
|------|-------------------|-------------|
| 포맷 | `"[추론 과정]\n...\n\n[분석 결과]\n..."` | `{"thought": "...", "analysis": "..."}` |
| 파싱 | `split("[분석 결과]")` — LLM 응답에 동일 문자열 포함 시 깨짐 | `json.loads()` — 구조적 파싱, 안정적 |
| 방어 | 없음 | `try/except` + 분리된 키 |
| 확장성 | 필드 추가 시 매직 스트링 추가 필요 | dict에 키 추가만 |

**에러 핸들링 전략**: reasoning LLM 실패(네트워크, API 제한 등) 시 메인 LLM(`self._llm`)으로 fallback 재시도. 메인도 실패하면 에러 메시지 반환. 이 경우 thinking 없이 프롬프트 엔지니어링만으로 추론.

## 6. 토큰 메트릭 처리

### 6.1 문제: reasoning LLM의 thinking 토큰 집계

reasoning 도구 내부에서 `self._llm_for_reasoning.invoke()`를 호출하면, 이 호출의 토큰 사용량은 **ToolNode가 자동 처리**하므로 그래프 상태에 직접 반영되지 않음.

### 6.2 현재 토큰 추적 경로

```
llm_node (line 528-536):
  response = self._llm_with_tools.invoke(context_messages)
  in_tokens = response.usage_metadata.get("input_tokens", 0)
  out_tokens = response.usage_metadata.get("output_tokens", 0)
```

llm_node는 **자신의 LLM 호출**만 추적. reasoning 도구 내부의 LLM 호출 토큰은 llm_node의 `usage_metadata`에 포함되지 않음.

### 6.3 해결 방안

reasoning 도구에서 토큰 정보를 JSON 반환에 포함:

```python
# reasoning 도구 내부
if isinstance(response.content, list):
    thought, text = extract_thought_from_content(response.content)
    result = {"thought": thought if show_thoughts else "", "analysis": text}
else:
    result = {"thought": "", "analysis": response.content}

# 토큰 정보 추가
if hasattr(response, "usage_metadata") and response.usage_metadata:
    result["reasoning_tokens"] = {
        "input": response.usage_metadata.get("input_tokens", 0),
        "output": response.usage_metadata.get("output_tokens", 0),
        "thinking": response.usage_metadata.get("output_token_details", {}).get("reasoning", 0),
    }

return json.dumps(result, ensure_ascii=False)
```

`_parse_result()`에서 `reasoning_tokens`를 추출하여 결과 dict에 포함. 이후 집계 경로:

```
_parse_result() → result["reasoning_tokens"] = {...}
  ↓
invoke() → return {**parsed, ...}  (reasoning_tokens가 그대로 전달)
  ↓
app.py handle_chat_message() → result에서 reasoning_tokens 추출
  ↓
기존 input_tokens/output_tokens에 합산하여 Message.input_tokens/output_tokens에 반영
```

**결정**: `Message` dataclass에 별도 `reasoning_tokens` 필드를 추가하지 않음. 기존 `input_tokens`/`output_tokens`에 합산하여 토큰 바에 반영하는 것이 UI 변경 최소화. `_parse_result()` 반환값에 `reasoning_tokens`를 포함하되, `invoke()`에서 기존 토큰에 합산:

```python
# invoke() 내부
reasoning_tokens = parsed.get("reasoning_tokens", {})
input_tokens += reasoning_tokens.get("input", 0)
output_tokens += reasoning_tokens.get("output", 0)
```

## 7. `show_thoughts` 일관성 보장

### 7.1 문제: 이중 경로 불일치 위험

`show_thoughts`가 두 곳에서 독립적으로 사용됨:

| 위치 | 역할 |
|------|------|
| `tools.py` reasoning 도구 클로저 | thought를 JSON에 포함할지 결정 |
| `react_graph.py` `_parse_result()` | JSON에서 thought를 추출할지 결정 |

두 값이 불일치하면: 도구는 thought를 포함했는데 파서가 무시하거나, 도구는 포함 안 했는데 파서가 찾으려고 시도.

### 7.2 해결: 단일 소스 (ReactGraphBuilder.show_thoughts)

두 경로 모두 `ReactGraphBuilder.show_thoughts`에서 유래:
- `build()` → `create_tools_with_services(show_thoughts=self.show_thoughts)`
- `_parse_result()` → `if self.show_thoughts and "reasoning" in tool_results`

**동일한 `self.show_thoughts` 값**이 양쪽에 전달되므로 불일치 불가. 이 전제를 테스트로 검증:

```python
def test_show_thoughts_consistency(self):

        mock_create.return_value = []
        with patch.object(ChatGoogleGenerativeAI, "bind_tools", return_value=MagicMock()):
```

## 12. 기대 효과

### Before (현재)

```
사용자: "안녕하세요" → casual_node (thinking 토큰 낭비)
사용자: "시간 알려줘" → llm_node → tool_node (thinking 토큰 낭비)
사용자: "A와 B 비교 분석해줘" → llm_node → reasoning tool (thinking + 프롬프트) → llm_node (thinking 낭비)
```

### After (변경 후)

```
사용자: "안녕하세요" → casual_node (thinking 없음, 빠름)
사용자: "시간 알려줘" → llm_node → tool_node (thinking 없음, 빠름)
사용자: "A와 B 비교 분석해줘" → llm_node → reasoning tool (thinking + 프롬프트, 고품질) → llm_node (thinking 없음)
```

| 항목 | Before | After |
|------|--------|-------|
| casual 응답 | thinking 토큰 소비 | 소비 없음 |
| 일반 도구 호출 | thinking 토큰 소비 | 소비 없음 |
| reasoning 호출 | thinking + 프롬프트 (전역) | thinking + 프롬프트 (**집중**) |
| reasoning 실패 시 | 에러 전파 | 메인 LLM fallback |
| 토큰 집계 | reasoning thinking 토큰 미포함 | JSON에 `reasoning_tokens` 포함 |
| 비용 효율 | 낮음 (전역 적용) | 높음 (필요 시에만) |
